# 让 AI 成为专业写作伙伴：Document Superpowers 方法论

# 引言：AI写作的痛点

你可能有过这样的体验：

打开 Claude 或 ChatGPT，说"帮我写一篇关于XX的文章"，AI 很快给出了一篇看起来像模像样的内容。但当你仔细读完，却发现：

- 开头就直奔主题，没有铺垫，像机器人在自说自话
- 逻辑跳跃，第三段突然冒出个没有前文铺垫的概念
- 结构松散，几个论点之间缺乏连贯性
- 读完了不知道核心观点是什么

于是你开始修改，"这段语气不对，重写"、"这部分展开太少，加点例子"、"这个逻辑有问题，重新组织一下"...来回三四轮，一个小时过去了，文章还是差点意思。

**问题的根源在哪？**

AI 不是写不好，而是缺乏**系统化的写作流程**。你直接让它"写"，它就真的只是"写"——没有前期思考、没有结构规划、没有质量把关。就像软件开发一样，不做需求分析和设计就直接撸代码，注定要返工。

那么，如果把软件开发的最佳实践应用到写作上，会怎么样？

# 方法论：四阶段工作流

**Document Superpowers** 就是这个答案。

它借鉴了开源项目 [obra/superpowers](https://github.com/obra/superpowers) 的软件开发方法论，将严谨的工程实践应用到写作流程中，形成了一套**四阶段系统化写作工作流**：

## 💡 Stage 1: Brainstorming（头脑风暴）

不是让 AI 直接写，而是先通过苏格拉底式提问理解主题：

- "目标读者是谁？"
- "你想让读者看完后做什么？"
- "核心观点用一句话怎么说？"

AI 一次问一个问题，逐步构建出一份**主题简报**（Topic Brief），明确：受众、目的、核心信息、语调风格。这个阶段有个**硬性门槛（HARD-GATE）**：不完成简报并获得你的批准，就不能进入下一阶段。

## 📋 Stage 2: Planning（规划大纲）

有了简报，AI 创建详细的文章大纲：

- 每个段落150-400字（bite-sized，易于管理）
- 明确每段的**目的、关键点、支撑证据**
- 规划段落间的**过渡衔接**
- 提前列出**研究清单**（避免写到一半发现缺材料）

大纲不是简单的标题列表，而是**可执行的写作蓝图**——就像软件开发的详细设计文档（SDD）。

## ✍️ Stage 3: Execution（执行写作）

按照大纲分段写作，提供两种模式：

- **批量模式**：写2-3段后暂停，给你检查（减少返工）
- **子代理模式**：每段启动一个新 AI 子会话，两轮审查（规格符合性 + 内容质量）

每段写完都有**自检清单**：字数达标？关键点覆盖？语调一致？过渡流畅？

## 🔍 Stage 4: Review（系统审查）

完成初稿后，进行**四遍独立审查**（每遍关注一个维度）：

1. **Logic Pass（逻辑）**：论证是否成立？有无矛盾？
2. **Evidence Pass（证据）**：每个断言都有支撑吗？来源可靠吗？
3. **Flow Pass（流畅度）**：段落间过渡自然吗？节奏合适吗？
4. **Polish Pass（润色）**：语法、格式、用词是否精准？

所有问题按**严重程度分类**（critical/moderate/minor），系统化修复后，产出 publication-ready 的最终文章。

---

这套流程的核心理念是：**增量验证 > 一次性输出**。每个阶段都有明确的产出和检查点，就像敏捷开发中的 sprint review。

# 价值：为什么需要系统化

看到这里，你可能会想："这么多流程，不是更慢了吗？"

来看个对比：

| 传统方式 | Document Superpowers |
|---------|---------------------|
| "写一篇关于XX的文章" | 四阶段系统化流程 |
| AI 一次性输出 3000 字 | 分段写作，每段验证 |
| 读完发现结构有问题 | 大纲阶段就明确结构 |
| 反复修改："这段重写"、"那段加点例子" | 每段写完自检，问题早发现 |
| 来回 3-4 轮，耗时 2-3 小时 | 40 分钟，publication-ready |

**表面上多了流程，实际上省了时间。**

## 三大核心价值

### 🎯 质量保证

四遍独立审查，每遍关注一个维度：
- **Logic Pass**：确保论证成立
- **Evidence Pass**：确保有理有据
- **Flow Pass**：确保阅读流畅
- **Polish Pass**：确保格式精良

传统方式是混在一起改，容易顾此失彼。分离关注点后，问题无处遁形。

### ⚡ 效率提升

**分段写作**减少了大段返工的风险。传统方式写完 3000 字才发现第二段逻辑有问题，前面引用它的部分全要改。Document Superpowers 在大纲阶段就能发现这种问题。

**检查点机制**让你在每 2-3 段后审查进度，及时纠偏，而不是写完整篇才发现跑偏。

### 📊 可追溯性

每个阶段都有产出文档：
- 主题简报：记录写作意图
- 详细大纲：记录规划思路
- 段落草稿：记录写作过程
- 审查日志：记录所有修改

这不仅方便复盘优化，也让协作写作变得可能——其他人能看到你的思考过程，而不只是最终文本。

---

说白了，**Document Superpowers 把"灵光一现的艺术"变成了"可复制的工程"**。不是每篇文章都能写好，但用了这套流程，至少能保证一个下限。

# 实践：快速上手指南

听起来不错？来试试吧！

## 📦 安装（30秒）

```bash
# 克隆到 OpenClaw skills 目录
git clone https://github.com/binbinao/document-superpowers.git \
  ~/.openclaw/skills/document-superpowers
```

安装完成后，技能会自动生效，无需额外配置。

## ✍️ 第一次使用

### Step 1: 告诉 AI 你要写什么

```
"我想写一篇关于 TypeScript 的博客文章"
```

AI 会自动调用 `brainstorming` 技能。

### Step 2: 回答问题（一次一个）

```
AI: "目标读者是谁？"
你: "JavaScript 开发者"

AI: "你希望读者看完后做什么？"
你: "尝试在新项目中使用 TypeScript"

AI: "核心观点用一句话怎么说？"
你: "TypeScript 能在编译时发现 bug，提升代码可靠性"
```

不用一次回答所有问题，AI 会一步步引导你。

### Step 3: 审阅主题简报

AI 会整理成一份简报，包含：受众、目的、核心信息、语调。**仔细看，这是后续写作的蓝图。**

### Step 4: 审阅文章大纲

AI 会生成详细大纲，每段 150-400 字，标注关键点、证据需求、过渡衔接。**这一步很关键，大纲清晰，写作就顺。**

### Step 5: 选择执行模式

- **批量模式**（推荐新手）：写 2-3 段后暂停，你审查，然后继续
- **子代理模式**（高级）：每段启动新 AI 会话，质量更高但稍慢

### Step 6: 在检查点审阅

每写完一批，AI 会暂停让你审查。**认真看！** 别只是点"继续"，这是纠偏的最佳时机。

### Step 7: 获得最终文章

四遍审查后，你会得到 publication-ready 的文章，外加一份详细的审查日志。

## 💡 小贴士

- **信任流程**：第一次可能觉得慢，但产出质量值得
- **认真审阅**：brainstorming 和 planning 阶段定调子，别敷衍
- **善用检查点**：发现问题立即说，别等全写完再大改
